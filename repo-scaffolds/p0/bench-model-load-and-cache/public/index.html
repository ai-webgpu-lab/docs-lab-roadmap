Cold vs Warm Model Load Harness

`bench-model-load-and-cache` uses a deterministic synthetic fixture to separate three phases: manifest fetch, model materialization, and prepared-artifact reuse. Cold mode clears Cache Storage and localStorage first. Warm mode reuses those artifacts when available.

The goal is not to mimic one framework perfectly. It is to make load and cache deltas observable before runtime-specific integrations land.

Scenario Controls

Run cold first to prime Cache Storage and prepared artifacts, then run warm to measure how much work gets skipped on the second visit.

Benchmark Guidance

  • Use the same browser session when comparing cold and warm results so storage state is meaningful.
  • Export one JSON draft per scenario. The UI delta is useful for comparison, but raw results should stay scenario-specific.
  • Replace the synthetic fixture with a real model manifest and weight preparation flow once the runtime choice is fixed.
